## 一、服务概述

### 1.1 服务定位

本服务是"应有成本估算系统"的智能客服模块，部署在门户网站（`portal/index.html`）右下角，以悬浮聊天窗口的形式为访客提供产品咨询、成本方法论解答等服务。

### 1.2 核心能力

- **产品问答**：解答关于系统功能、使用方式、定价策略等问题
- **成本方法论**：解释应有成本估算的公式、工艺参数、行业标准等专业知识
- **试用引导**：当问题超出知识范围时，引导用户提交表单联系人工客服
- **智能语义理解**：支持用户用不同表述方式提问，系统自动理解意图

### 1.3 技术栈

| 组件 | 技术 | 版本 | 用途 |
|------|------|------|------|
| Web 框架 | FastAPI | ≥0.109 | 提供 REST API 服务 |
| 服务器 | Uvicorn | ≥0.27 | ASGI 服务器 |
| 向量数据库 | ChromaDB | ≥0.4.22 | 存储 FAQ 语义向量 |
| Embedding 服务 | SiliconFlow API | - | 文本向量化（语义理解） |
| 大语言模型 | DeepSeek API | deepseek-chat | 兜底回答生成 |
| 配置管理 | python-dotenv | ≥1.0 | 环境变量管理 |

### 2.2 与芋道后端的关系

| 系统 | 职责 | 通信关系 |
|------|------|---------|
| **AI 服务** (`ai-service/`) | 智能客服问答 | 独立运行，与芋道后端无直接调用 |
| **门户网站** (`portal/index.html`) | 产品展示 + AI 助手前端 | 直接调用 AI 服务 API |
| **芋道后端** (`costmodel_backend/`) | 业务系统、表单数据存储 | 仅接收门户网站的表单提交 |

**关键设计：AI 服务与芋道后端完全解耦**

- AI 服务是独立的 Python 进程，不依赖芋道后端的任何接口或数据库
- 即使芋道后端停机，AI 客服仍然可以正常工作
- 这种设计使得 AI 服务可以独立部署、独立扩容、独立升级

## 三、核心运行机制

### 3.1 一次问答的完整流程

当用户在门户网站输入一个问题，比如 **"材料费用怎么计算"**，系统内部经历了以下步骤：

#### 第 1 步：接收请求

```
用户输入: "材料费用怎么计算"
    ↓
门户网站 JS 发送 fetch 请求:
    POST http://localhost:8082/chat/ask
    Body: {"question": "材料费用怎么计算"}
    ↓
FastAPI 接收请求 → Pydantic 验证 → 调用 faq_engine.answer()
```

#### 第 2 步：文本向量化（Embedding）

```
faq_engine.answer() 调用 vector_store.search()
    ↓
APIEmbeddingStore.search() 被调用
    ↓
调用 SiliconFlow Embedding API:
    POST https://api.siliconflow.cn/v1/embeddings
    Model: BAAI/bge-large-zh-v1.5
    Input: "材料费用怎么计算"
    ↓
返回 1024 维向量: [0.023, -0.156, 0.089, ...]
```

**什么是 Embedding？**

Embedding 是将人类语言转换为计算机能理解的数学向量的过程。具有相似含义的句子，在向量空间中的距离会很近。

例如：
- "材料费用怎么计算" → 向量 A
- "直接材料成本怎么计算" → 向量 B
- A 和 B 在向量空间中的距离很近（cosine 相似度 0.79），所以系统知道它们是同一个意思

这就是"换种说法也能理解"的技术原理。

#### 第 3 步：向量检索

```
将用户问题的向量 与 ChromaDB 中存储的所有 FAQ 向量进行比对
    ↓
ChromaDB 使用 cosine 距离算法，返回最相似的 Top-3 条 FAQ:

┌──────────────────────────────────────────────────────────────┐
│ 排名 │ 匹配到的问题                    │ 相似度 │ 答案摘要  │
├──────────────────────────────────────────────────────────────┤
│  1   │ 直接材料成本怎么计算？          │ 0.7901 │ 公式：... │
│  2   │ 材料单价应包含哪些费用？        │ 0.6148 │ 包含：... │
│  3   │ 期间费用有哪些？费率如何确定？   │ 0.6140 │ 包含：... │
└──────────────────────────────────────────────────────────────┘
```

#### 第 4 步：阈值判断与响应

```
最高相似度 = 0.7901
    ↓
判断: 0.7901 ≥ 0.75 (FAQ_THRESHOLD_HIGH)
    ↓
结论: 高置信命中 → 直接返回答案
    ↓
返回结果:
{
    "answer": "直接材料成本 = 材料用量 × 材料单价 - 废料回收量 × 废料单价...",
    "source": "faq",
    "confidence": 0.7901,
    "matched_question": "直接材料成本怎么计算？"
}
```

### 3.2 三级响应策略详解

系统根据相似度阈值，采用三种不同的响应策略：

#### 策略一：高置信直接命中（相似度 ≥ 0.75）

**触发条件**：用户问题与某条 FAQ 高度相似

**处理方式**：直接返回预设答案，不调用任何外部 LLM API

**成本**：¥0（不消耗 API 额度）

**响应速度**：< 500ms

**适用场景**：80% 以上的标准问题

**示例**：
```
用户: "什么是应有成本"
匹配: "什么是应有成本？" 相似度 0.9507
结果: 直接返回预设答案（source: faq）
```

#### 策略二：半相关 LLM 生成（0.40 ≤ 相似度 < 0.75）

**触发条件**：用户问题与知识库有一定关联，但没有完全匹配的答案

**处理方式**：将检索到的 FAQ 作为上下文，调用 DeepSeek API 进行合理推断和生成

**成本**：消耗 DeepSeek API tokens（每次约 ¥0.01~0.05）

**响应速度**：1~3 秒

**适用场景**：用户换了一种新说法，或问题涉及多个知识点的综合

**示例**：
```
用户: "废品费用包含哪些内容"
匹配: "废品费用如何计入应有成本？" 相似度 0.7196
处理: 将"废品费用如何计入应有成本"的答案作为上下文
      调用 DeepSeek 生成："根据知识库，废品费用包含..."
结果: source: llm，基于上下文推断的回答
```

**为什么这样做？**

如果不调用 LLM，直接返回相似度 0.7196 的 FAQ 答案，用户可能会觉得"答非所问"（因为问题是"包含哪些内容"，而 FAQ 是"如何计入"）。通过 LLM 基于上下文重新组织语言，回答更精准、更自然。

#### 策略三：低相似度拒绝（相似度 < 0.40）

**触发条件**：用户问题与知识库完全无关

**处理方式**：直接拒绝，不调用 LLM，避免幻觉

**成本**：¥0

**响应速度**：< 500ms

**返回话术**：
> "抱歉，这个问题超出了我的知识范围。您可以点击页面右下角的【立即体验】提交表单，我们的专业顾问将在 24 小时内为您详细解答。"

**示例**：
```
用户: "你们公司在哪里"
匹配: "利润应该定多少？" 相似度 0.396
结果: 直接拒绝（source: reject）
```

**为什么拒绝而不是让 LLM 自由发挥？**

制造业成本数据要求**绝对准确**。一个错误的公式可能让客户在供应商谈判中出丑。与其让模型"编"一个似是而非的答案，不如诚实地说"这超出我的知识范围"。

### 3.3 为什么每个用户提问都要调用 Embedding API？

这是必须的，因为：

1. **用户问题每次都不同**：即使问同一个主题，措辞也可能不同（"材料费咋算" vs "直接材料成本怎么计算"）
2. **必须实时向量化**：只有将用户问题转成向量，才能在向量空间中与 FAQ 向量进行相似度比较
3. **Embedding API 费用极低**：每次调用约 10~50 tokens，按 SiliconFlow 价格约 ¥0.5/百万 tokens，相当于几千次提问才花几毛钱

## 四、设计思想与决策原因

### 4.1 为什么采用 RAG 架构，而不是直接让 LLM 回答？

#### ❌ 直接让 LLM 回答的问题

| 问题 | 说明 |
|------|------|
| **幻觉风险** | LLM 可能会编造不存在的公式、错误的行业数据 |
| **专业准确性** | 制造业成本估算有严格的公式和标准，LLM 训练数据中未必准确 |
| **成本不可控** | 每个问题都调 LLM，费用是 RAG 方案的 3~5 倍 |
| **无法审计** | 无法追溯某个回答的来源，出问题无法定位 |

#### ✅ RAG 架构的优势

| 优势 | 说明 |
|------|------|
| **答案可控** | 80% 的问题直接返回人工维护的 FAQ，保证 100% 准确 |
| **消除幻觉** | 低相似度直接拒绝，绝不编造 |
| **成本极低** | 高置信命中不消耗 LLM API，大幅节省费用 |
| **可审计** | 每个答案都能追溯到具体 FAQ 条目 |
| **可更新** | 运营人员编辑 JSON 文件即可更新知识 |

### 4.2 为什么 Embedding 用 SiliconFlow，而不是本地模型？

#### 方案对比

| 方案 | 代表 | 优点 | 缺点 |
|------|------|------|------|
| **本地模型** | sentence-transformers / BAAI/bge-small-zh | 零网络依赖、零 API 费用 | 需要下载模型（~100MB~1GB）、需要 GPU/CPU 资源、模型质量一般 |
| **API 服务** | **SiliconFlow** | 模型质量高（bge-large-zh-v1.5）、无需本地资源、按量付费极便宜 | 需要网络、需要 API Key |

#### 选择 SiliconFlow 的原因

1. **模型质量好**：`BAAI/bge-large-zh-v1.5` 是中文 Embedding 领域的标杆模型，语义理解能力强
2. **费用极低**：Embedding API 几乎是所有 LLM 服务中最便宜的，实际使用几乎可以忽略成本
3. **无需维护模型**：不用关心模型下载、版本更新、显存占用等问题
4. **国内访问稳定**：SiliconFlow 是国内服务，网络延迟低，访问稳定

### 4.3 为什么 FAQ 数据用 JSON 文件维护，而不是数据库？

#### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **JSON 文件** | 简单直观、版本可控（git）、编辑方便、无需数据库运维 | 不适合超大规模数据（>1万条）、没有并发写入能力 |
| **MySQL/PostgreSQL** | 支持复杂查询、事务、并发 | 增加运维复杂度、需要表结构设计 |
| **专用知识库** | 功能强大、可视化编辑 | 引入额外系统、学习成本高 |

#### 选择 JSON 的原因

1. **当前规模合适**：30 条 FAQ 用 JSON 完全够用，预计到 500 条以内都不需要换方案
2. **运营人员友好**：产品经理/运营可以直接打开文件编辑，不需要懂 SQL
3. **版本可追溯**：JSON 文件纳入 git 管理，谁改了什么一目了然，改错了可以回滚
4. **部署简单**：不需要安装和配置数据库服务

### 4.4 为什么 AI 服务要与芋道后端解耦？

1. **技术栈差异**：芋道后端是 Java（Spring Boot），AI 服务是 Python，解耦避免技术栈强绑定
2. **独立演进**：AI 能力可以快速迭代升级，不受后端发布节奏限制
3. **独立扩缩容**：AI 服务可以单独部署更多实例应对流量高峰
4. **多场景复用**：同一套 AI 服务可以同时服务门户网站、小程序、APP 等多个前端
5. **故障隔离**：芋道后端出问题时，AI 客服仍然可用，反之亦然

## 五、FAQ 数据维护指南

### 5.1 FAQ 数据结构

```json
[
  {
    "question": "什么是应有成本？",
    "answer": "应有成本（Should Be Cost）是指一项产品或服务在现有技术与市场条件下，理应达到的成本水平..."
  },
  {
    "question": "直接材料成本怎么计算？",
    "answer": "直接材料成本 = 材料用量 × 材料单价 - 废料回收量 × 废料单价..."
  }
]
```

### 5.2 维护流程

#### 日常维护（添加/修改/删除 FAQ）

**方法一：直接编辑 JSON 文件（推荐）**

```bash
# 1. 编辑 FAQ 数据文件
vi ai-service/data/faq.json

# 2. 重新导入到向量库
cd ai-service
python init_faq.py

# 3. 验证导入成功
curl http://localhost:8082/faq/count
```

**方法二：调用 API 热更新（不重启服务）**

```bash
# 单条添加
curl -X POST http://localhost:8082/faq/add \
  -H "Content-Type: application/json" \
  -d '{"question":"新问题","answer":"新答案"}'

# 批量覆盖导入（会先清空旧数据）
curl -X POST http://localhost:8082/faq/seed \
  -H "Content-Type: application/json" \
  -d '{"faqs":[{"question":"问题1","answer":"答案1"}]}'
```

### 5.3 FAQ 编写规范

#### 问题（question）编写原则

1. **口语化**：尽量模拟用户真实提问方式
   - ✅ "直接材料成本怎么计算？"
   - ❌ "论直接材料成本核算方法论"

2. **覆盖多种问法**：同一个知识点，尽量写多条不同问法
   - "什么是应有成本？"
   - "应有成本是什么意思？"
   - "请解释一下应有成本"

3. **包含关键词**：确保包含用户可能搜索的关键词
   - "材料利用率是什么？行业基准大概多少？"

4. **避免过于宽泛**：问题要具体
   - ✅ "零基估算的六步法是什么？"
   - ❌ "你们系统怎么用？"（太宽泛，难以准确回答）

#### 答案（answer）编写原则

1. **简洁专业**：控制在 200 字以内，直接回答核心
2. **包含公式**：涉及计算的，务必给出公式
   ```
   直接材料成本 = 材料用量 × 材料单价 - 废料回收量 × 废料单价
   ```
3. **引用数据**：涉及行业数据的，给出参考范围
   ```
   行业数据参考：钣金冲压60%-75%，注塑成型95%-98%
   ```
4. **结构清晰**：使用分段、编号等方式让答案易读

### 5.4 FAQ 数据补充方向（建议）

根据用户常见咨询场景，建议逐步补充以下方向的 FAQ：

## 七、常见问题排查

### Q1: 服务启动报错 `ModuleNotFoundError: No module named 'xxx'`

**原因**：依赖包未安装

**解决**：
```bash
cd ai-service
pip install -r requirements.txt
```

### Q2: 问答返回 `source: "reject"`，明明有对应的 FAQ

**原因**：相似度低于阈值（0.40）

**排查步骤**：
1. 检查 FAQ 问题表述是否太正式，与用户口语化提问差异大
2. 尝试降低 `FAQ_THRESHOLD_LOW`（如从 0.40 调到 0.30）
3. 补充更多同义问法的 FAQ 条目

### Q3: 回答匹配到了不相关的 FAQ

**原因**：相似度过高阈值（0.75）设置偏低，或 FAQ 之间区分度不够

**解决**：
1. 提高 `FAQ_THRESHOLD_HIGH`（如从 0.75 调到 0.80）
2. 优化 FAQ 表述，让不同问题的语义差异更明显

### Q4: Embedding API 调用失败

**排查**：
```bash
# 测试 SiliconFlow API 连通性
curl https://api.siliconflow.cn/v1/models \
  -H "Authorization: Bearer $EMBEDDING_API_KEY"

# 检查 .env 中 API Key 是否正确
# 检查 SiliconFlow 控制台余额是否充足
```

**兜底**：将 `EMBEDDING_MODE` 改回 `local`，系统会自动降级为本地字符串匹配

### Q5: 门户网站 AI 助手显示"服务暂时不可用"

**排查步骤**：
1. 检查 AI 服务是否运行：`curl http://localhost:8082/`
2. 检查门户网站中的 API 地址是否正确：
   ```javascript
   // portal/index.html 中
   var AI_API_URL = 'http://localhost:8082/chat/ask';
   ```
3. 检查浏览器控制台是否有 CORS 报错（AI 服务已配置 `allow_origins=["*"]`）
4. 检查防火墙是否拦截了 8082 端口
